== Context and Scope

In diesem Kapitel wird das System gegenüber externen Akteuren und angebundenen Nachbarsystemen abgegrenzt.
Es wird beschrieben, welche Rolle das Python-System einnimmt, welche externen Komponenten beteiligt sind
und über welche Schnittstellen die Kommunikation erfolgt.

Das betrachtete System ist die Python-basierte Steuerungssoftware. Sie stellt eine einheitliche API bereit,
über die Benutzer Roboterbewegungen, Werkzeugbefehle und Weltbefehle auslösen können. Je nach Betriebsmodus
werden diese Befehle an die Unity-Simulation, an den realen Roboterarm oder an beide Zielsysteme weitergeleitet.

=== Business Context

Der Business Context beschreibt die Nutzung des Systems aus Sicht der Benutzer, zum Beispiel Studierende
oder Robotik-Interessierte. Im Vordergrund steht dabei nicht die technische Umsetzung, sondern der typische
Ablauf der Nutzung.

Der Benutzer schreibt Python-Code und verwendet dabei die bereitgestellte API. Das System übernimmt die
interne Verarbeitung, prüft Eingaben und leitet Befehle an die Simulation (Unity) oder an den realen Roboterarm weiter.
Dadurch muss der Benutzer nicht direkt mit Unity, UDP-Nachrichten, serieller Kommunikation oder der
Herstellerbibliothek `pymycobot` arbeiten.

=== Technical Context

Der technische Kontext zeigt die Systemgrenzen sowie die Kommunikationswege zwischen den einzelnen Komponenten.

Im Zentrum steht die Python-API, die als Fassade des Systems dient. Sie nimmt Befehle aus dem Benutzerprogramm
entgegen und kapselt die interne Weiterleitung. Für die Simulation werden JSON-Nachrichten über UDP an Unity
gesendet. Für die reale Hardware wird die Herstellerbibliothek `pymycobot` verwendet, welche über eine serielle
USB-Verbindung mit dem Roboter kommuniziert.

Das folgende Diagramm zeigt die technischen Schnittstellen und die wichtigsten internen Komponenten.

| Schnittstelle / Komponente | Protokoll / Art | Zweck

| Lokal (Python)
| Bildet die Fassade des Systems.
Sie stellt die zentrale Benutzerschnittstelle bereit und kapselt die interne Steuerung, Moduswahl und Weiterleitung der Befehle.
Dabei werden die Befehle anhand des eingestellten Betriebsmodus weitergeleitet.

| **mypalletizer**
| Internes Python-Modul
| Verarbeitet Befehle für die Simulation und stellt die Verbindung zur Unity-Simulation her.
Darüber werden sowohl Roboterbefehle als auch Weltbefehle an Unity weitergegeben.

| **pymycobot**
| Python-Bibliothek
| Wird intern verwendet, um den realen MyPalletizer 260 M5 anzusteuern.
Die Bibliothek ist gekapselt, sodass der Benutzer nicht direkt mit ihr arbeiten muss.

| **Simulations-Roboterkanal**
| UDP (JSON), Port `5005`
| Überträgt Steuerbefehle vom System an die Unity-Simulation.
Dazu gehören zum Beispiel Gelenkbewegungen, Werkzeugwahl, Greiferzustand oder Pumpensteuerung.

| **Simulations-Weltkanal**
| UDP (JSON), Port `5006`
| Überträgt Weltbefehle vom System an die Unity-Simulation.
Dazu gehören zum Beispiel das Erzeugen, Löschen oder Zurücksetzen von Objekten in der Simulationswelt.

| **Hardware-Kanal**
| Serial (USB), `115200` Baud
| Überträgt Steuerbefehle über `pymycobot` an den realen MyPalletizer 260 M5.

==== Abgrenzung des Systems

Innerhalb der Systemgrenze befinden sich die Python-API sowie die internen Module zur Steuerung und Weiterleitung
der Befehle. Dazu gehören insbesondere die Logik zur Moduswahl, die Erzeugung der Nachrichten und die Anbindung
an Simulation oder Hardware.

Nicht Teil des Systems sind die Unity-Anwendung selbst und der physische Roboterarm. Beide werden als externe
Zielsysteme betrachtet, die vom Python-System angesteuert werden.

Die Benutzer arbeiten nicht direkt mit den internen Modulen, UDP-Sockets oder `pymycobot`, sondern ausschliesslich
mit der bereitgestellten Python-API.

==== Betriebsmodi

Das System unterstützt drei Betriebsmodi:

* **`virtual`**
  Befehle werden nur an die Unity-Simulation gesendet.

* **`real`**
  Befehle werden nur an den realen Roboterarm gesendet.

* **`both`**
  Befehle werden sowohl an die Unity-Simulation als auch an den realen Roboterarm gesendet.

Die Moduswahl erfolgt innerhalb der Python-API. Dadurch bleibt der Benutzercode weitgehend gleich, unabhängig
davon, ob Simulation, Hardware oder beide Zielsysteme verwendet werden.

==== Beispielhafte JSON-Nachrichten

Die Kommunikation zwischen Python und Unity erfolgt über strukturierte JSON-Nachrichten.
Dabei wird zwischen Roboterbefehlen und Weltbefehlen unterschieden. Beide Nachrichtentypen
verwenden eine gemeinsame Grundidee: Jede Nachricht enthält eine Protokollversion `v` sowie
ein Feld `type`, das den Befehl eindeutig beschreibt.

Die konkreten Nachrichten werden auf Python-Seite durch das Modul `protocol` erzeugt.
Dadurch bleibt der Aufbau der übertragenen Daten zentral definiert und konsistent.

.Beispiel einer Nachricht für eine asynchrone Gelenkbewegung
[source,json]
----
{
  "v": 1,
  "type": "move_joints",
  "j1": 0.0,
  "j2": 20.0,
  "j3": -15.0,
  "j4": 10.0,
  "speed": 40
}
----

Diese Nachricht wird über den Roboterkanal an die Simulation übertragen und beschreibt den
Soll-Zustand der vier Gelenke sowie die gewünschte Bewegungsgeschwindigkeit.

.Beispiel einer Nachricht für die Werkzeugwahl
[source,json]
----
{
  "v": 1,
  "type": "set_end_effector",
  "tool": "gripper"
}
----

Diese Nachricht wählt das aktive Werkzeug des Roboters. Unterstützt werden insbesondere
`gripper` und `pump`.

.Beispiel einer Nachricht für das Erzeugen eines Objekts in der Simulationswelt
[source,json]
----
{
  "v": 1,
  "type": "spawn",
  "object_type": "box",
  "name": "box_1",
  "position": {
    "x": 0.2,
    "y": 0.0,
    "z": 0.1
  },
  "rotation": {
    "x": 0.0,
    "y": 0.0,
    "z": 0.0
  },
  "scale": {
    "x": 0.05,
    "y": 0.05,
    "z": 0.05
  },
  "color": {
    "r": 255,
    "g": 0,
    "b": 0
  }
}
----

Diese Nachricht wird über den Weltkanal übertragen und beschreibt ein neues Objekt, das in
Unity erzeugt werden soll. Neben Typ und Name enthält sie auch Transformations- und
Darstellungsparameter wie Position, Rotation, Skalierung und Farbe.

Die Beispiele zeigen bewusst nur einen kleinen Ausschnitt des Protokolls. Weitere Befehle
wie `sync_move_joints`, `led`, `set_gripper_state`, `pump`, `delete` und `clear` folgen
demselben Grundprinzip.
